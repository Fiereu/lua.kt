# lua.kt

Guidance for AI coding agents working in this repository. This file is the single
source of truth. Tool-specific files (e.g. `CLAUDE.md`) link back to it.

## Project layout

- Kotlin/JVM project built with Gradle (Kotlin DSL).
- Dependencies and plugin versions are declared in the version catalog at
  `gradle/libs.versions.toml`. Add new ones there, not inline.
- Three modules. `runtime` is the library: lexer, AST, parser, evaluator, and the
  standard library, all `internal`, with only the curated `de.fiereu.lua` API
  (plus `de.fiereu.lua.common.LuaBytes`) public. It uses Kotlin explicit-API
  strict mode and is publishable via `maven-publish`. `repl` is the demo app: an
  interactive shell that also runs a script file when given a path
  (`repl/src/main/kotlin/de/fiereu/lua/repl/Main.kt`). `benchmarks` is a small
  profiling harness comparing reference `lua`, lua.kt on a fresh runtime, and
  lua.kt on a warmed-up runtime. Add further modules as sibling directories and
  `include(...)` them in `settings.gradle.kts`.
- Shared build logic lives in `buildSrc/` as precompiled convention plugins.

## Common commands

Always use the Gradle wrapper (`./gradlew`):

- Build: `./gradlew build`
- Run the REPL: `./gradlew :repl:run`
- Run a script: `./gradlew :repl:run --args="path/to/script.lua"`
- Profile against reference Lua: `./gradlew :benchmarks:profile` (add
  `-Plua=/path/to/lua` to point the reference leg at a specific binary)
- Publish the library locally: `./gradlew :runtime:publishToMavenLocal`
- Run tests: `./gradlew test`
- Check formatting: `./gradlew spotlessCheck`
- Auto-format: `./gradlew spotlessApply`
- Run all checks: `./gradlew check`

## Conventions
- Kotlin and the Java 25 toolchain are configured by the
  `kotlin-conventions` plugin in `buildSrc/`.
- Tests use kotest (wired up by the `test-conventions` plugin).
  Put tests under `src/test/kotlin`.
- Code is formatted with Spotless/ktlint. Run `./gradlew spotlessApply` before
  committing.
- SonarLint static analysis runs as part of `./gradlew check`.

## Writing style

Write prose that reads like a person wrote it.

- No em dashes (—). Split into two sentences with a period, or use a comma.
- No semicolons in prose. Split into two sentences.
- No single-character symbols like →. Use the multi-character equivalent such as `->`.

Hyphens in compound words such as non-null are fine.

## Comments

Comment only when the reader needs something the code cannot tell them. Hidden
constraints, subtle invariants, bug workarounds, or behavior that would surprise
a reader.

Do not comment to:

- Restate what well-named identifiers already say.
- Reference the current task or fix. That belongs in the PR description.
- Justify a design decision unless it is non-obvious from the surrounding code.

No visual section-separating comments.

## Commits and branches

- Use [Conventional Commits](https://www.conventionalcommits.org) for messages,
  for example `feat: add user export` or `fix(parser): handle empty input`.
  Common types are `feat`, `fix`, `docs`, `refactor`, `test`, and `chore`.
- Name branches conventionally too, as `type/short-description` in kebab-case,
  for example `feat/user-export`.
- Do not add any branding to commits, code, or docs. No tool or vendor names,
  no "generated by" lines, no co-author trailers for a tool, and no promotional
  emojis or badges.

## Before committing

- Make sure `./gradlew check` passes.
- Make sure the code is formatted (`./gradlew spotlessApply`).
